# ADR-0003: 라이선스 정책 — 순수 영감 (Pure Inspiration, No Coroot Code Touch)

| 메타 | 값 |
|---|---|
| **상태** | Accepted |
| **결정일** | 2026-05-18 (재결정) |
| **결정자** | 사용자 (P0 단계 의사결정) |
| **관련 ADR** | ADR-0001 (K8s 버전 정책), ADR-0002 (K8s feature 상태표) |
| **관련 문서** | `CLAUDE.md` §2, `docs/10-benchmarks/coroot-mapping.md` §4 |
| **법무 검토 필요 여부** | **❌ 불필요** — 본 결정은 AGPL 의무를 트리거하지 않는 운영 방식을 명시적으로 채택 |

---

## 1. 컨텍스트 (Context)

본 프로젝트는 [Coroot](https://github.com/coroot/coroot)(AGPLv3)을 벤치마크하는 On-Premise 옵저버빌리티 플랫폼이다.

이전 결정 초안(하이브리드 — `experiments/`에서 Coroot 코드 활용 + 클린룸 절차)을 검토한 결과 다음과 같은 부담이 식별되었다:
- 사내 법무 검토를 반드시 받아야 함 (D-7 데드라인 압박)
- `experiments/` 격리·build tag·CI license-scan·SBOM 등 운영 가드레일이 다수 필요
- 클린룸 절차의 단일 엔지니어 환경 강제력이 약함
- 사후 감사 발생 시 "experiments/ 격리가 충분했는가" 입증 부담

본 ADR은 **"AGPL 라이선스 코드를 어떤 형태로든 본 프로젝트의 저장소·작업 흐름에 들이지 않는다"** 는 더 단순하고 안전한 정책으로 재결정한다.

---

## 2. 결정 (Decision)

### 2.1 핵심 원칙: **Pure Inspiration**
- **Coroot 코드(.go/.c/.yaml/.sql 등 어떤 소스 파일이든)를 본 저장소에 두지 않는다.**
- **Coroot 코드를 로컬에 clone/fork하지 않는다.** (반드시 봐야 할 때만 GitHub 웹 UI에서 일시 열람 — 다운로드 금지)
- **본 프로젝트는 Coroot의 "아이디어·컨셉·아키텍처"만 참조한다.** 코드 표현은 참조 대상이 아니다.
- 결과: AGPL이 "전이"될 가능성이 **구조적으로 0** → 법무 검토 자체가 불필요.

### 2.2 지식 습득 허용 원천 (Whitelist)
다음 원천만 사용하며, 본 프로젝트의 모든 학습·조사는 여기서 출발한다.

| 원천 | 라이선스 / 위험도 | 활용 |
|---|---|---|
| **Coroot 공식 문서** (`docs.coroot.com`) | 일반 문서 라이선스, 학습 자유 | 아키텍처 / 기능 / 설치 / 운영 컨셉 |
| **Coroot 공식 블로그** (`coroot.com/blog`) | 동일 | 디자인 결정·이유 |
| **Coroot 컨퍼런스 발표** (KubeCon 등 공개 영상) | 발표 자료 보통 자유 | 깊은 통찰 |
| **Coroot GitHub Issues / Discussions** | 사실관계 인용 자유 | 문제·해결 사례 |
| **학술 논문** (Drain 알고리즘 등) | 학술 공개 | 알고리즘 직접 구현 |
| **eBPF 공식 문서** (libbpf, cilium/ebpf) | 다양 — 학습 자유 | eBPF 패턴 |
| **Linux Kernel 문서 / man pages** | 학습 자유 | kernel API |
| **OpenTelemetry spec** | Apache-2.0 | 직접 채택 |
| **Prometheus / ClickHouse 공식 문서** | 학습 자유 | 통합 패턴 |
| **CNCF 표준 / KEP / RFC** | 공개 표준 | K8s 통합 |

### 2.3 금지 사항 (MUST NOT)
- ❌ Coroot 저장소를 `git clone` / `git submodule` / `git subtree`로 본 저장소에 포함
- ❌ Coroot의 소스 파일(`.go`, `.c`, `.yaml`, `.proto`, `.sql` 등)을 본 저장소에 복사 — 변수명 변경·번역·refactor 등 어떤 가공도 무관
- ❌ Coroot 코드 주석·에러 메시지·도구 출력 문구의 번역·인용
- ❌ Coroot 코드를 보면서 동시에 본 저장소 코드를 작성 (시간적·공간적 분리)
- ❌ 본 프로젝트의 모든 산출물(코드/매니페스트/문서)에 AGPL 라이선스를 채택

### 2.4 `experiments/` 역할 재정의
이전 결정 초안에서 도입했던 `experiments/`는 **AGPL 학습 용도가 아닌 일반 R&D 용도로만** 존재한다.
- 자체 알고리즘 PoC, 성능 비교 실험, kernel 동작 검증 등에는 사용 OK
- **Coroot 또는 다른 AGPL 코드는 절대 두지 않는다** (다른 모든 디렉터리와 동일 규칙)
- 빌드 격리(`//go:build experiments`) 같은 특수 가드레일은 **불필요** — 그냥 일반 디렉터리

### 2.5 `reference/` 디렉터리 — **삭제** (불필요)
이전 결정 초안의 `reference/coroot-upstream/`는 본 저장소에 두지 않는다.
- Coroot 코드를 봐야 할 매우 드문 경우(예: 특정 KEP 동작 검증)는 GitHub 웹 UI에서 일시 열람
- 로컬 클론 금지

### 2.6 본 프로젝트의 SPDX 라이선스
- **AGPL을 채택하지 않는다** (위 §2.3에 의해 자동)
- 구체 라이선스(Apache-2.0 / MIT / Proprietary 등)는 **별도 ADR(0003-1)에서 사내 정책에 따라 결정**

### 2.7 운영 가드레일 (단순화)
이전 초안의 복잡한 가드레일(experiments build tag, CI license-scan, SBOM AGPL 검출)은 **대부분 불필요**해진다.

| 가드레일 | 이전 (하이브리드) | 현재 (순수 영감) |
|---|---|---|
| `experiments/` build tag 강제 | 필요 | **불필요** |
| `.dockerignore`/`.helmignore` experiments 제외 | 필요 | **불필요** |
| AGPL 코드 스캔 (`scancode` 등) | 필요 | **간이만 필요** — 우연한 AGPL 의존성 도입 차단용 |
| SBOM 발행 | AGPL 검출용 필수 | **표준 보안 관점만** (일반 OSS 컴플라이언스) |
| PR 클린룸 체크박스 | 필요 | **불필요** — 클린룸 자체 없음 |
| PR template AGPL 체크 | 4개 항목 | **1개로 단순화**: "이 PR에 AGPL 라이선스 코드를 도입하지 않았다" |

### 2.8 매핑표 (`coroot-mapping.md`) Tier 일괄 강등
ADR-0003에 따라 **모든 항목의 채택 깊이를 다음과 같이 강제**:

| 기존 표기 | 강제 변환 |
|---|---|
| 🟢 Copy → | **🔵 Inspire** (코드 카피 금지 = Copy 불가) |
| 🟡 Adapt → | **🔵 Inspire** (코드 차용 금지 = Adapt 불가) |
| 🔵 Inspire | 유지 |
| 🔴 Replace | 유지 |
| ⚫ Drop | 유지 |

매핑표의 "변경 사항" 컬럼은 **공식 문서/논문에서 학습한 컨셉을 어떻게 우리 도메인에 맞게 적용할 것인가**로 의미가 재정의된다.

## 3. 결과 (Consequences)

### 3.1 긍정적
- **법무 검토 불필요** — 본 결정 자체가 AGPL 트리거 조건을 회피
- **사내 정책 충돌 위험 0** — 회사가 어떤 OSS 정책을 갖든 통과
- **운영 가드레일 대폭 단순화** (build tag, 클린룸 등 모두 제거)
- **사외 배포 100% 자유** — 라이선스 선택지 무제한
- **사후 감사 부담 0** — 입증해야 할 격리·절차 없음

### 3.2 부정적 / 비용
- **개발 속도 감소** — 코드 직접 참고 불가로 디테일 디버깅 시 비용 증가 (추정 +40~60%)
- **공식 문서가 부족한 영역에서 정보 빈곤** — 예: eBPF L7 파서의 세부 구현은 Coroot 코드에 더 풍부하지만 우리는 접근 불가
- **재발견(Re-discovery) 비용** — Coroot이 이미 해결한 엣지케이스를 우리가 다시 만남

### 3.3 완화책
- 공식 문서 → 학술 논문 → 커뮤니티 표준 → eBPF 공개 예제(MIT/Apache 라이선스 프로젝트들) 순으로 학습 경로 우선순위화
- 재발견 비용은 자체 테스트로 흡수 (테스트 커버리지 강화)
- 막히면 사용자에게 보고 — Coroot 코드 열람이 정말 필요한지 다시 판단 (열람만, 다운로드 금지)

## 5. 검토된 대안 (Alternatives Considered)

| 경로 | 결정 시점 | 결과 |
|---|---|---|
| A: AGPLv3 수용 (본 프로젝트도 AGPL) | 1차 평가 | 기각 — 사외 배포 제약 + 사내 OSS 정책 충돌 위험 |
| B: 완전 클린룸 (Coroot 코드 안 봄) | 1차 평가 | 기각 — R&D 속도 손실 과대 |
| C: EE 듀얼 라이선스 협상 | 1차 평가 | 보류 — 비용·시간 불확실 |
| **D: 하이브리드** (experiments/ 격리 + 클린룸) | 1차 채택 → 재검토 후 기각 | **기각** — 법무 검토 필수 + 운영 가드레일 부담 |
| **E: 순수 영감 (Pure Inspiration)** | **2차 채택** | **현재 결정** — 법무 부담 0, 가드레일 최소 |

> 본 ADR은 1차 채택이었던 D(하이브리드)를 supersede한다.
> 1차 채택 사유와 재검토 사유는 본 ADR의 §1 컨텍스트와 사용자 의사결정 기록에 보존.

## 6. 실무 가이드 — "아이디어를 어떻게 가져오는가"

### 6.1 OK: 공식 문서·논문·표준에서 학습
- "Coroot은 Drain 알고리즘으로 로그 클러스터링" → IEEE 논문(He et al., 2017)에서 직접 학습 → 자체 구현
- "Coroot은 eBPF의 `tcp_v4_connect` kprobe로 TCP 연결 추적" → libbpf 공식 문서·kernel man page에서 학습 → 자체 구현
- "Coroot은 service map 식별자로 (namespace, workload, container) 튜플 사용" → 자체 결정으로 동일 채택 (사실 자체는 저작권 보호 대상 아님)
- "Coroot은 L7 프로토콜을 user-space에서 파싱" → 아키텍처 결정 채택. 파싱 로직은 RFC/공식 spec에서 학습 후 자체 구현

### 6.2 NG: 코드를 보면서 작성
- ❌ Coroot의 `parser.c`를 GitHub 웹에서 열어두고 우리 `parser.go`를 작성
- ❌ Coroot의 함수 시그니처를 그대로 옮기고 내부만 다시 작성
- ❌ Coroot의 테스트 케이스 데이터를 카피
- ❌ Coroot의 에러 메시지를 한국어로 번역해서 사용

### 6.3 회색 영역 처리 원칙
의심스러우면 **포기**한다 — 다른 출처에서 정보를 구하거나, 자체 시행착오로 해결. 사후에 "그때 Coroot 코드를 봤었나?"가 문제가 되지 않도록 한다.
